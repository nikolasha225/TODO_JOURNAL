## Обновлённый список обязательных требований к приложению Todo (с графическим интерфейсом)

### 1. Функциональность ядра
- Хранение данных в формате **JSON**.
- Класс `TodoJournal` с атрибутами:
  - `path_todo` – путь к файлу журнала.
  - `name` – название журнала.
  - `entries` – список задач.
- Методы класса:
  - `create(filename: str, name: str) -> None` – статический метод для создания нового пустого журнала.
  - `add_entry(new_entry: str) -> None` – добавление задачи.
  - `remove_entry(index: int) -> None` – удаление задачи по индексу.
  - `edit_entry(index: int, new_text: str) -> None` – редактирование задачи.
  - `_parse() -> dict` – чтение и парсинг JSON-файла (приватный).
  - `_update(new_data: dict) -> None` – запись данных в файл (приватный).
  - `get_name() -> str` – возвращает название журнала.
  - `get_tasks() -> list` – возвращает список задач.
- Магические методы:
  - `__len__()` – возвращает количество задач.
  - `__iter__()` – итератор по задачам (реализован через `yield`).
  - `__getitem__(index)` – доступ к задаче по индексу (поддерживает срезы).
  - `__getattr__(name)` – динамический доступ к атрибутам `first` и `last`.
  - `__setattr__(name, value)` – защита от изменения `first` и `last` (readonly).

### 2. Графический интерфейс (GUI)
- **Главное окно**:
  - Отображает название текущего журнала.
  - Список задач в виде таблицы или списка (например, `QListWidget`, `QTableWidget`, `tkinter.Listbox`).
  - Кнопки для основных операций: **Добавить**, **Удалить**, **Редактировать**, **Обновить** (перечитать файл).
  - Возможность выбора журнала (если их несколько) через выпадающий список или отдельное окно настроек.
- **Диалоги**:
  - При добавлении задачи – диалог ввода текста (с возможностью указать дату/время).
  - При удалении – подтверждение (сообщение с вопросом).
  - При редактировании – диалог с текущим текстом задачи.
- **Индикация статуса**: статусная строка или всплывающие уведомления об успешных операциях и ошибках.
- **Настройки**: отдельное окно для изменения конфигурации (путь к журналу по умолчанию, редактор для редактирования задач, формат даты и т.д.).
- **Многопоточность** (рекомендуется): длительные операции (например, работа с Gitlab API) не должны блокировать интерфейс; используются `QThread` (PyQt) или `threading` с соответствующим обновлением UI.

### 3. Обработка ошибок
- При отсутствии файла журнала – вывод сообщения пользователю через диалог и предложение создать новый журнал.
- При некорректном формате JSON – вывод сообщения и возможность исправить (например, открыть файл во внешнем редакторе или восстановить из резервной копии).
- Собственное исключение `TodoJournalError` для специфических ошибок (проблемы с конфигурацией, повреждённый файл).

### 4. Конфигурация
- Конфигурационный файл в формате **YAML**, расположенный в каталоге, соответствующем XDG Base Directory (`~/.config/todo/config.yaml`).
- Содержимое:
  - путь к текущему журналу (или список журналов с именами);
  - редактор по умолчанию (для редактирования задач);
  - настройки GUI (размер окна, последний выбранный журнал, тема оформления);
  - формат отображения даты.
- При отсутствии конфига – создание дефолтного (путь к журналу по умолчанию, стандартные настройки).

### 5. Логирование
- Настроено через конфигурационный файл (например, `logging.ini`) или программно.
- **Уровни логирования**:
  - В GUI показываются только ошибки (в диалогах).
  - В файл записываются все сообщения (`DEBUG` и выше), включая действия пользователя (добавление, удаление, редактирование, ошибки, события GUI).
- Формат сообщений в файле: `время [модуль:уровень] - сообщение` (с указанием даты-времени, имени модуля, уровня и текста).
- **Ротация логов**: ежедневная (`TimedRotatingFileHandler`), хранение за последние 7 дней.

### 6. Тестирование
- Используется **pytest**.
- Тесты покрывают:
  - логику класса `TodoJournal` (создание, добавление, удаление, редактирование, магические методы);
  - обработку исключений (отсутствие файла, повреждённый JSON);
  - работу с конфигурацией (чтение/запись YAML);
  - **GUI-тесты** (с помощью `pytest-qt` для PyQt или `pytest-tkinter` для Tkinter) – проверка отображения элементов, реакция на клики, диалоги.
- Фикстуры:
  - `tmpdir` для временных файлов;
  - собственные фикстуры для предварительно заполненных журналов;
  - фикстуры для создания экземпляров GUI (например, `qapp` для PyQt).
- Mocking (`monkeypatch`) используется для подмены ввода/вывода, имитации внешних зависимостей.
- **Покрытие кода** (coverage) должно быть высоким (рекомендуется 90%+ для логики, GUI тесты могут снижать процент из-за сложности, но бизнес-логика должна быть покрыта полностью).

### 7. Документация
- **Docstring** для всех публичных и приватных методов (в формате, принятом в проекте, с указанием параметров, возвращаемых значений и исключений).
- **README.md** в корне репозитория, содержащий:
  - описание проекта;
  - скриншоты интерфейса;
  - инструкцию по установке (включая `poetry`);
  - описание основных возможностей;
  - инструкцию по запуску тестов.
- Возможно, отдельная документация по пользовательскому интерфейсу.

### 8. Стиль кода и инструменты
- Код соответствует **PEP8** (проверяется `pylint`).
- Используются **аннотации типов** (из модуля `typing`) для всех функций и методов (кроме некоторых частей GUI, где типы могут быть неочевидны).
- Git-практики:
  - работа через **issues** и **ветки**;
  - коммиты с осмысленными сообщениями;
  - использование `stash`, `cherry-pick`, интерактивного `rebase`.

### 9. Упаковка и установка
- Проект упакован с помощью **poetry** (файл `pyproject.toml`).
- После установки доступен исполняемый скрипт (например, `todo-gui`), запускающий графическое приложение.
- Возможна сборка в standalone-приложение (например, с помощью `pyinstaller` или `cx_Freeze`) для распространения без установки Python.

## Список библиотек (зависимости)

### Стандартные (входят в Python)
- `json`
- `sys`
- `os`
- `logging` (включая `logging.config`, `logging.handlers`)
- `textwrap`
- `typing`
- `io`
- `secrets`
- `string`
- `threading` (если не используется `QThread`)

### Сторонние (требуют установки)

#### Для GUI (выбрать одну из опций)
1. **PyQt5** / **PyQt6** – мощный, многофункциональный, есть инструменты для тестирования (`pytest-qt`).  
   - `PyQt5` (лицензия GPL) или `PyQt6` (аналогично).  
   - Рекомендуется для серьёзных проектов.

2. **PySide6** – официальные биндинги от Qt for Python, лицензия LGPL.  
   - Аналогично по функциональности, но с более свободной лицензией.

3. **Tkinter** – встроенный в Python, простой, но менее гибкий.  
   - Подходит для небольших проектов и быстрого прототипирования.

4. **wxPython** – ещё один вариант, менее распространённый, но тоже полнофункциональный.

#### Для тестирования GUI (если выбраны PyQt/PySide)
- **pytest-qt** – интеграция pytest с Qt.

#### Для работы с конфигурацией
- **PyYAML** – парсинг YAML.
- **python-xdg** – поддержка XDG Base Directory.

#### Для интеграции с Gitlab (опционально)
- **python-gitlab** – работа с Gitlab API.
- **urllib3** – для отключения предупреждений SSL.

#### Для статического анализа и тестирования
- **pytest** – тестирование.
- **pytest-cov** – измерение покрытия (опционально).
- **coverage** – измерение покрытия.
- **pylint** – анализ кода.

#### Для упаковки и управления зависимостями
- **poetry** – управление зависимостями и упаковка.
- **pyinstaller** – создание standalone-исполняемых файлов (опционально).

### Примечания
- Для тестирования GUI необходимо настроить виртуальный дисплей (например, `xvfb` в CI), если тесты запускаются без графической среды.
- Выбор GUI-библиотеки должен быть обоснован: PyQt/PySide дают больше возможностей для сложного интерфейса (таблицы, вкладки, стилизация), Tkinter – минималистичен и не требует дополнительных зависимостей. В учебных целях можно выбрать Tkinter для простоты, но с учётом того, что в курсе изучались и более сложные инструменты, PyQt предпочтительнее для отработки навыков.
