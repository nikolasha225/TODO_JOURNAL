## План создания todo-приложения с GUI (поэтапно, помодульно)

### 0. Общая структура проекта (финальный вид)

```
todo-gui/
├── .gitignore
├── .coveragerc
├── pyproject.toml               # poetry
├── README.md
├── logging.ini                  # конфигурация логирования
├── src/
│   ├── __init__.py
│   ├── todo_journal.py          # класс TodoJournal
│   ├── config.py                # работа с конфигурацией YAML
│   ├── exceptions.py            # собственные исключения
│   └── gui/
│       ├── __init__.py
│       ├── main_window.py       # главное окно
│       ├── dialogs.py           # диалоги (добавить, редактировать, настройки)
│       └── app.py               # запуск приложения

│   ├── __init__.py

```

## Этап 1: Создание базовой структуры и настройка окружения

**Задача:** Инициализировать репозиторий, настроить poetry, создать файлы конфигурации.

**Файлы:**  
- `pyproject.toml` – зависимости (pytest, coverage, pylint, pyyaml, python-xdg, а также библиотека для GUI, например, tkinter – встроенная, так что не требует записи). Для тестирования GUI можно добавить `pytest-qt`, но если используется Tkinter, можно обойтись без.
- `.gitignore` – стандартный для Python, включая `__pycache__/`, `.coverage`, `logs/`.
- `.coveragerc` – настройка coverage (исключить тесты, `__init__.py`, `gui/`? – но нужно покрывать GUI, хотя это сложнее).
- `logging.ini` – конфигурация логирования (как в PZ8).
- `README.md` – пока заглушка.

**Действия:**
1. Создать каталог проекта, инициализировать git.
2. Выполнить `poetry init` и добавить зависимости.
3. Создать структуру каталогов `src/`, `tests/`, `logs/`.
4. Добавить пустые `__init__.py` в `src/` и `tests/`.

## Этап 2: Реализация ядра – класс TodoJournal (аналог PZ3-PZ5)

**Задача:** Написать класс для работы с todo-журналом: чтение, запись, добавление, удаление, редактирование, магические методы. Покрыть тестами.

**Файлы:**  
- `src/todo_journal.py` – класс `TodoJournal`.
- `tests/test_todo_journal.py` – тесты.

**Содержание `todo_journal.py`** (кратко):
- Конструктор: принимает путь к файлу, вызывает `_parse()`, заполняет `self.name` и `self.entries`.
- Методы: `create` (статический), `add_entry`, `remove_entry`, `edit_entry`, `_parse`, `_update`.
- Магические: `__len__`, `__iter__` (yield), `__getitem__`, `__getattr__` (first, last), `__setattr__` (readonly).
- Обработка ошибок: при отсутствии файла – поднимать `TodoJournalError` (позже реализуем).

**Тесты:**
- Использовать фикстуру `tmpdir` для временных файлов.
- Проверить создание журнала, добавление/удаление/редактирование записей.
- Проверить работу магических методов.
- Проверить обработку ошибок (несуществующий файл, повреждённый JSON).

## Этап 3: Реализация конфигурации и логгирования (аналог PZ7-PZ8)

**Задача:** Создать модуль для работы с конфигурацией (YAML) и настроить логирование.

**Файлы:**  
- `src/exceptions.py` – класс `TodoJournalError` (с разными типами ошибок, форматированием через `textwrap`).
- `src/config.py` – функции для получения пути к конфигу (`get_config_path`), загрузки/сохранения конфигурации (использовать `xdg.BaseDirectory`).
- `logging.ini` – уже создан на этапе 1, нужно настроить (два хэндлера: консольный (только ERROR) и файловый (все уровни) с ротацией).
- `tests/test_config.py` – тесты конфигурации (проверка создания дефолтного конфига, чтение/запись).

**Примечание:** В GUI консольный лог может не понадобиться, но в файл всё равно пишем.

## Этап 4: Реализация GUI (главное окно и диалоги)

**Задача:** Создать графический интерфейс с помощью Tkinter (или PyQt). Реализовать:
- главное окно со списком задач;
- кнопки: добавить, удалить, редактировать, обновить, настройки;
- диалоги: добавление/редактирование, подтверждение удаления;
- отображение статуса (строка состояния);
- выбор текущего журнала (если несколько).

**Выбор GUI:** Для простоты и встроенности – Tkinter. Для более профессионального подхода – PyQt5. Я опишу Tkinter, но можно адаптировать.

**Файлы:**  
- `src/gui/__init__.py` – экспорт основного класса.
- `src/gui/main_window.py` – класс главного окна.
- `src/gui/dialogs.py` – классы диалогов.
- `src/gui/app.py` – функция `main()` для запуска приложения.

**Содержание `main_window.py`**:
- Класс `MainWindow(tk.Tk)` или `tk.Toplevel`.
- Атрибуты: `journal` (объект `TodoJournal`), `listbox` (или `treeview`).
- Методы: `load_journal`, `refresh_list`, `add_entry`, `delete_entry`, `edit_entry`, `open_settings`.
- Использовать `tkinter.simpledialog` для ввода текста, `tkinter.messagebox` для подтверждения/ошибок.

**Содержание `dialogs.py`**:
- `AddEditDialog(tk.Toplevel)` – поле ввода, кнопки OK/Cancel.
- `SettingsDialog(tk.Toplevel)` – настройки (путь к журналу, редактор, формат даты). Загружает/сохраняет конфиг.

**Содержание `app.py`**:
- Импортирует `MainWindow`.
- В функции `main()`: загружает конфигурацию, получает путь к текущему журналу, создаёт объект `TodoJournal` (обрабатывая возможные ошибки), запускает главное окно.

## Этап 5: Интеграция GUI с ядром и конфигурацией

**Задача:** Связать GUI с объектом `TodoJournal` и конфигурацией.

**Файлы:**  
- `src/gui/main_window.py` (дополнить).
- `src/gui/app.py` (дополнить).

**Действия:**
1. В `app.py` после загрузки конфига инициализировать `TodoJournal` и передать его в `MainWindow`.
2. В `MainWindow` реализовать методы, вызывающие соответствующие методы `journal`:
   - `add_entry` – вызвать `journal.add_entry` и обновить список.

   - `edit_entry` – показать диалог с текущим текстом, затем вызвать `journal.edit_entry`.
   - При ошибках (например, повреждённый JSON) показывать `messagebox.showerror`.
3. Настройки: при изменении конфига (например, пути к журналу) перезагружать журнал и обновлять список.

## Примечания по реализации GUI

- **Выбор Tkinter:** Простой, не требует дополнительных библиотек, но менее гибкий. Для диалогов используем `tkinter.simpledialog` и `tkinter.messagebox`. Для списка – `tkinter.Listbox`. Обновление списка – очистка и повторное добавление.
- **Выбор PyQt:** Более мощный, требует установки `PyQt5` или `PySide6`. Позволяет использовать `QListView` с моделью, легко добавлять колонки (например, дату). Тестирование через `pytest-qt` удобно.

В данном плане я ориентируюсь на Tkinter как на минимально зависимый вариант, но в реальном проекте я бы рекомендовал PyQt из-за лучшей поддержки тестирования и более современного внешнего вида.

Если вы решите использовать PyQt, то:
- В `pyproject.toml` добавить `PyQt5` или `PySide6`.
- В тестах использовать `pytest-qt`.
- В GUI использовать `QMainWindow`, `QListWidget`, `QDialog` и т.д.
