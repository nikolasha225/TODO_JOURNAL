## Список обязательных требований к приложению Todo

### 1. Функциональность
- Хранение данных в формате **JSON**.
- Класс `TodoJournal` с атрибутами:
  - `path_todo` – путь к файлу журнала.
  - `name` – название журнала.
  - `entries` – список задач.
- Методы класса:
  - `create(filename: str, name: str) -> None` – статический метод для создания нового пустого журнала.
  - `add_entry(new_entry: str) -> None` – добавление задачи.
  - `remove_entry(index: int) -> None` – удаление задачи по индексу.
  - `_parse() -> dict` – чтение и парсинг JSON-файла (приватный).
  - `_update(new_data: dict) -> None` – запись данных в файл (приватный).
  - `get_name() -> str` – возвращает название журнала.
  - `get_tasks() -> list` – возвращает список задач.
- Магические методы:
  - `__len__()` – возвращает количество задач.
  - `__iter__()` – итератор по задачам (реализован через `yield`).
  - `__getitem__(index)` – доступ к задаче по индексу (поддерживает срезы).
  - `__getattr__(name)` – динамический доступ к атрибутам `first` и `last`.
  - `__setattr__(name, value)` – защита от изменения `first` и `last` (readonly).

### 2. Обработка ошибок
- При отсутствии файла журнала – выход из программы с кодом 1 и сообщением об ошибке.
- При некорректном формате JSON – вывод сообщения пользователю.
- Собственное исключение `TodoJournalError` для специфических ошибок (например, проблемы с конфигурацией, повреждённый файл).

### 3. CLI (интерфейс командной строки)
- Реализован с использованием модуля `argparse`.
- Поддерживаются команды:
  - Создание журнала.
  - Добавление задачи (передаётся текст задачи).
  - Удаление задачи по индексу (с интерактивным подтверждением).
  - Редактирование задачи (открытие редактора, заданного в конфиге).
  - Выбор активного журнала при наличии нескольких.
- Параметры:
  - `--delete <index>` – удалить задачу.
  - `--edit <index>` – отредактировать задачу.
- При отсутствии аргументов – переход в интерактивный режим (показ задач, добавление).

### 4. Конфигурация
- Конфигурационный файл в формате **YAML**, расположенный в каталоге, соответствующем XDG Base Directory (`~/.config/todo/todo_config.yaml`).
- Содержимое:
  - путь к текущему журналу (или список журналов с именами);
  - редактор по умолчанию (например, `nano`, `vim`);
  - другие настройки (например, формат даты).
- При отсутствии конфига – создание дефолтного (путь к журналу по умолчанию, предложение создать новый).
- Поддержка нескольких журналов: при запуске можно указать имя журнала для работы.

### 5. Логирование
- Настроено через конфигурационный файл (например, `logging.ini`).
- **Уровни логирования**:
  - В консоль выводятся только сообщения уровня `ERROR` и выше.
  - В файл записываются все сообщения (`DEBUG` и выше), включая действия пользователя (добавление, удаление, редактирование, ошибки).
- Формат сообщений в файле: `время [модуль:уровень] - сообщение` (с указанием даты-времени, имени модуля, уровня и текста).
- **Ротация логов**:
  - Ежедневная ротация (`TimedRotatingFileHandler`).
  - Логи хранятся за последние 7 дней.

### 6. Тестирование
- Используется **pytest**.
- Тесты покрывают:
  - создание журнала;
  - добавление и удаление задач;
  - магические методы (`__len__`, `__iter__`, `__getitem__`, `__getattr__`);
  - обработку исключений (отсутствие файла, повреждённый JSON);
  - CLI (с подменой `sys.argv` или через `monkeypatch`);
  - работу с конфигурацией и временными файлами.
- Фикстуры:
  - `tmpdir` для временных файлов;
  - собственные фикстуры для предварительно заполненных журналов.
- Mocking (`monkeypatch`) используется для:
  - подмены ввода/вывода;
  - имитации внешних зависимостей (например, Gitlab API).
- **Покрытие кода** (coverage) должно быть 100% для основного кода (исключая тесты и служебные файлы).

### 7. Документация
- **Docstring** для всех публичных и приватных методов (в формате, принятом в проекте, с указанием параметров, возвращаемых значений и исключений).
- **README.md** в корне репозитория, содержащий:
  - описание проекта;
  - инструкцию по установке (включая `poetry`);
  - примеры использования CLI;
  - инструкцию по запуску тестов.

### 8. Стиль кода и инструменты
- Код соответствует **PEP8** (проверяется `pylint`).
- Используются **аннотации типов** (из модуля `typing`) для всех функций и методов.
- Git-практики:
  - работа через **issues** и **ветки**;
  - коммиты с осмысленными сообщениями;
  - использование `stash`, `cherry-pick`, интерактивного `rebase`.

### 9. Упаковка и установка
- Проект должен быть упакован с помощью **poetry** (файл `pyproject.toml`).
- После установки должен быть доступен исполняемый скрипт (например, `todo`), запускающий CLI.

## Список библиотек (зависимости)

### Стандартные (входят в Python)
- `json`
- `sys`
- `os`
- `argparse`
- `logging` (включая `logging.config`, `logging.handlers`)
- `textwrap`
- `typing`
- `io`
- `secrets`
- `string`

### Сторонние (требуют установки)
- **pytest** – тестирование
- **pytest-cov** – измерение покрытия (опционально, но рекомендовано)
- **coverage** – измерение покрытия
- **pylint** – анализ кода
- **PyYAML** – работа с YAML-конфигами
- **python-xdg** – поддержка XDG Base Directory
- **python-gitlab** – для интеграции с Gitlab (если используется)
- **urllib3** – для отключения предупреждений SSL (при работе с Gitlab)
- **poetry** – управление зависимостями и упаковка (не требуется в коде, но используется для сборки)

### Примечание
- Для работы с Gitlab (в рамках курса) может потребоваться библиотека `python-gitlab`, но она не является обязательной для базового функционала.
